# 架构说明

## 总体目标

本项目采用一个 TypeScript monorepo，支撑宠物急救分诊、医院联系、导航和到院前协同。

当前仅有工程骨架和目录边界，尚未初始化可运行工程。

## Monorepo 结构

```text
.
├── apps/
│   ├── miniapp/              # 微信小程序/H5，用户 SOS 急救入口
│   ├── hospital-admin/       # 医院后台，接收 SOS 队列并回拨
│   └── ops-admin/            # 运营后台，医院资料和数据质检
├── services/
│   └── api/                  # NestJS 后端 API
├── packages/
│   ├── shared-types/         # 前后端共用类型
│   ├── api-client/           # 前端请求 SDK
│   ├── safety-rules/         # 红旗规则和医疗安全常量
│   └── config/               # 共享配置
└── docs/
    ├── 架构说明.md
    ├── 接口文档.md
    ├── 安全要求.md
    └── 分支与回滚.md
```

## 用户端架构

用户端第一版只围绕 SOS 急救主流程：

1. 首页大 SOS 按钮。
2. SOS 极简表单。
3. 定位授权或手动选择位置。
4. 医院匹配结果。
5. 提交成功页。
6. 保持电话畅通提示。

用户端不做：

- 红黄绿风险等级展示。
- AI 医生聊天。
- 用药建议。
- 诊断结果。
- 复杂社区或商城。

## 医院端架构

医院后台第一版只做 SOS 接收与处理：

1. SOS 队列。
2. SOS 详情。
3. 一键回拨。
4. 接听状态记录。
5. 医生备注。
6. 是否建议到院。
7. 案例关闭。

医院后台可以看到后台红旗标记，但该标记只用于优先处理，不作为对用户展示的诊断结果。

## 后端模块

后端未来按模块拆分：

| 模块 | 作用 |
|---|---|
| `auth` | 用户、医院、运营身份认证 |
| `pet-profile` | 宠物档案 |
| `sos-intake` | SOS 提交与表单版本 |
| `hospital-matching` | 医院匹配和距离排序 |
| `hospital-console` | 医院队列、回拨和备注 |
| `case-timeline` | 急救案例事件时间线 |
| `safety-rules` | 后台红旗标记和禁用话术边界 |
| `anonymization` | 脱敏训练样本导出 |
| `provider-map` | 地图供应商适配 |
| `provider-notification` | 微信通知、短信、电话能力适配 |
| `ai-adapter` | 后续 AI 摘要与训练接口，MVP 默认关闭 |

后续业务逻辑必须按模块归属修改，避免把 SOS、医院、权限、AI 和脱敏逻辑混在一起。

## 核心数据对象

| 对象 | 说明 |
|---|---|
| `User` | 宠主账号 |
| `PetProfile` | 宠物档案 |
| `Hospital` | 医院资料 |
| `HospitalStaff` | 医院人员 |
| `SosCase` | SOS 急救案例 |
| `SosFormSnapshot` | 用户提交时的表单快照 |
| `HospitalHandling` | 医院回拨和处理记录 |
| `CaseTimelineEvent` | 案例时间线事件 |
| `SafetyFlag` | 后台红旗标记 |
| `AiTrainingSample` | 脱敏后的 AI 训练样本 |

每条 SOS 记录必须包含 `schemaVersion`，用于后续增加字段时兼容旧数据。

## SOS 主流程

```text
用户打开小程序
↓
点击首页大 SOS 按钮
↓
回答三个核心问题
↓
系统补充手机号、定位、宠物档案和提交时间
↓
系统匹配附近可接收 SOS 的医院
↓
医院后台收到 SOS 队列
↓
医生回拨用户并电话指导
↓
医院记录处理结果
↓
平台沉淀结构化数据
```

SOS 必填问题控制在三个以内：

| 问题 | 选项 |
|---|---|
| 宠物类型/品种 | 猫、狗、其他，可补充品种 |
| 目前状态 | 呼吸异常、抽搐、出血、误食、外伤、尿不出、精神差、其他 |
| 血型 | A、B、AB、DEA、未知 |

血型允许选择“未知”，不得阻塞提交。

## 扩展点

第一版骨架必须保留以下扩展点：

- 表单版本：SOS 问题以后可以增加，但旧记录不能坏。
- Provider 接口：地图、通知、电话、AI、训练导出都可以替换供应商。
- 事件时间线：任何新增动作都写成事件，不硬塞进单个状态字段。
- 权限边界：用户、医院、运营、内部开发权限分开。
- 数据脱敏层：训练 AI 只能使用脱敏数据。
- 医疗安全层：红旗规则和禁用话术独立维护。

## 移动端扩展路径

MVP 先做微信小程序，不创建原生 iOS 或 Android 工程。

未来 App 应复用：

- `packages/shared-types`
- `packages/api-client`
- `services/api`
- SOS 流程设计

如果进入原生 App 或 React Native 阶段，必须单独制定技术方案、成本评估、审核标准和数据安全方案。

## UI 原则

小程序第一屏必须突出大 SOS 急救按钮。

用户最慌的时候，路径必须最短：

- 入口清晰。
- 问题尽量少。
- 文案不制造误导。
- 提交后提醒保持电话畅通。
- 不展示红黄绿风险等级。

旧的四宫格 UI 演示已从长期文档入口移除，后续以正式小程序页面设计为准。

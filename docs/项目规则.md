# 项目规则

本文档定义宠物急救分诊与到院前协同平台的核心规则，适用于整个项目生命周期。

---

## 一、项目定位

**本项目不是 AI 宠物医生。**

本平台定位为：**急救信息收集与到院前协同工具**。

核心价值：
- 缩短宠物主人在紧急情况下的决策时间
- 让医院在宠物到院前提前掌握关键信息
- 支持医生通过电话在到院前指导紧急处置
- 为后续 AI 辅助能力积累结构化数据基础

---

## 二、MVP 功能边界

### 必须实现（MVP）

| 功能 | 说明 |
|------|------|
| SOS 按钮 | 首页核心入口，一键触发急救流程 |
| 简化问卷 | 不超过 5 个必填问题，采集症状关键信息 |
| 自动信息补全 | 自动读取联系方式、定位、宠物档案、时间戳 |
| 信息发送 | 将 SOS 表单发送至医院后台 |
| 医生回拨支持 | 后台显示待处理 SOS，支持医生电话指导 |

### 不在 MVP 范围内

- AI 诊断或风险评估（对用户端）
- 在线支付或挂号
- 宠物档案完整管理（MVP 阶段仅读取，不提供完整编辑）
- 多院竞价或派单
- 视频问诊

---

## 三、用户角色

| 角色 | 说明 |
|------|------|
| 宠物主人 | 触发 SOS、填写基本信息、等待医生回拨 |
| 医院接诊员 | 接收 SOS 信息、分配给医生、记录处理状态 |
| 执业医生 | 查看 SOS 详情、电话指导、录入处理记录 |
| 平台管理员 | 管理医院账号、监控系统运行 |

---

## 四、数据规则

### 4.1 自动采集字段

以下字段由系统自动补全，无需用户手动填写：

- 用户联系方式（来自注册信息）
- 地理定位（来自设备 GPS）
- 宠物档案基本信息（来自已绑定的宠物档案）
- 提交时间戳

### 4.2 用户必填字段（≤5 项）

以下字段需要用户在 SOS 流程中快速作答：

1. 主要症状描述（文字或快速选择）
2. 症状持续时间
3. 宠物当前意识状态（清醒 / 昏迷 / 不确定）
4. 是否有出血或外伤（是 / 否）
5. 目标医院（系统推荐 + 用户确认）

### 4.3 红旗标记规则

- 系统根据症状关键词自动判断是否标记红旗
- 红旗标记**仅在医院后台可见**
- 用户端不展示任何风险等级（红/黄/绿）
- 红旗仅用于医院内部优先处理队列排序

### 4.4 数据隐私

- 宠物主人信息按敏感数据处理
- 脱敏规则：姓名、手机号、地址均需脱敏后方可用于分析
- AI 训练数据必须经用户授权且完成脱敏

---

## 五、医疗安全边界（强制）

### 系统不得

- 诊断疾病或作出任何医学判断
- 开具处方或推荐任何药物
- 提供药物名称、剂量或用法
- 向用户承诺宠物安全或预后
- 建议用户在出现红旗症状时等待或延迟就医
- 以任何形式让 AI 替代执业医生指导急救

### 系统可以

- 收集和传递结构化的急救基本信息
- 将 SOS 信息推送至医院后台
- 支持医生回拨并记录指导内容
- 自动生成病例摘要（由医生核准后使用）
- 在后台保留红旗标记
- 脱敏后沉淀数据用于未来 AI 能力建设

---

## 六、技术规则（工程阶段适用）

以下规则在进入工程阶段后生效：

- 用户端：Taro + React + TypeScript（微信小程序）
- 后台管理：Next.js + TypeScript
- 后端 API：NestJS + TypeScript
- 数据库 ORM：Prisma
- Monorepo 结构：`apps/`、`services/`、`packages/`
- 所有接口必须有类型定义，禁止使用 `any`
- 所有涉及用户隐私的 API 必须鉴权

---

## 七、内容与语言规范

- 用户端文案使用简体中文
- 文案语气：简洁、冷静、不制造恐慌
- 禁止在用户端出现"危险""死亡""无法救治"等负面词语
- 所有提示文案须经医疗顾问审核后上线

---

## 八、版本与发布规则

- 规划文档阶段：文档变更无需版本号
- 工程阶段：遵循语义化版本（Semantic Versioning）
- 任何功能上线前须完成内部测试和医疗安全审查

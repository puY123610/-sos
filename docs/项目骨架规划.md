# 项目骨架规划

## 1. 目标

本文件只描述未来工程项目应该如何搭建，不代表当前已经创建代码工程。

当前阶段先把项目骨架规划清楚，等用户确认进入开发阶段后，再用微信开发者工具、Taro、NestJS、Next.js、Prisma 等正式初始化项目。

## 2. 总体骨架

未来正式项目采用 TypeScript monorepo：

## 3. 用户端骨架

用户端第一版只围绕 SOS 急救主流程：

1. 首页大 SOS 按钮。
2. SOS 极简表单。
3. 定位授权或手动选择位置。
4. 医院匹配结果。
5. 提交成功页。
6. 保持电话畅通提示。

用户端不做：

- 红黄绿风险等级展示
- AI 医生聊天
- 用药建议
- 诊断结果
- 复杂社区或商城

## 4. 医院端骨架

医院后台第一版只做 SOS 接收与处理：

1. SOS 队列。
2. SOS 详情。
3. 一键回拨。
4. 接听状态记录。
5. 医生备注。
6. 是否建议到院。
7. 案例关闭。

医院后台可以看到后台红旗标记，但该标记只用于优先处理，不作为对用户展示的诊断结果。

## 5. 后端模块骨架

后端未来按模块拆分：

| 模块 | 作用 |
|---|---|
| auth | 用户、医院、运营身份认证 |
| pet-profile | 宠物档案 |
| sos-intake | SOS 提交与表单版本 |
| hospital-matching | 医院匹配、距离排序 |
| hospital-console | 医院队列、回拨、备注 |
| case-timeline | 急救案例事件时间线 |
| safety-rules | 后台红旗标记 |
| anonymization | 脱敏训练样本导出 |
| provider-map | 地图供应商适配 |
| provider-notification | 微信通知、短信、电话能力适配 |
| ai-adapter | 后续 AI 摘要与训练接口，MVP 默认关闭 |

## 6. 数据骨架

核心数据对象：

| 对象 | 说明 |
|---|---|
| User | 宠主账号 |
| PetProfile | 宠物档案 |
| Hospital | 医院资料 |
| HospitalStaff | 医院人员 |
| SosCase | SOS 急救案例 |
| SosFormSnapshot | 用户提交时的表单快照 |
| HospitalHandling | 医院回拨和处理记录 |
| CaseTimelineEvent | 案例时间线事件 |
| SafetyFlag | 后台红旗标记 |
| AiTrainingSample | 脱敏后的 AI 训练样本 |

每条 SOS 记录必须包含 `schemaVersion`，用于后续增加字段时兼容旧数据。

## 7. 扩展预留

为了后续好加功能，第一版就要保留以下扩展点：

- 表单版本：SOS 问题以后可以增加，但旧记录不能坏。
- Provider 接口：地图、通知、电话、AI、训练导出都可以替换供应商。
- 事件时间线：任何新增动作都写成事件，而不是硬塞到单个状态字段。
- 权限边界：用户、医院、运营、内部开发权限分开。
- 数据脱敏层：训练 AI 只能用脱敏数据。
- 医疗安全层：红旗规则和禁用话术独立维护。

## 8. 开发顺序建议

正式进入开发后，建议顺序：

1. 初始化 monorepo 和基础工程配置。
2. 建立 shared-types。
3. 建立 Prisma 数据模型。
4. 实现 SOS 创建 API。
5. 实现医院 SOS 队列 API。
6. 实现小程序 SOS 页面。
7. 实现医院后台 SOS 队列页。
8. 加权限测试、红旗标记测试、脱敏导出测试。
9. 再考虑 AI、通知、运营后台和更多商业化功能。

# SOS 急救实施规划

本文档详细描述宠物急救分诊与到院前协同平台中 SOS 急救流程的实施规划。

---

## 一、SOS 流程总览

```
用户点击 SOS 按钮
       ↓
进入 SOS 流程，立即进行症状快速填写（≤5项）
       ↓
系统自动补全：定位 + 联系方式 + 宠物档案 + 时间戳
       ↓
用户确认目标医院
       ↓
提交 SOS（一键发送）
       ↓
后台接收，红旗自动标记
       ↓
医院接诊员分配给医生
       ↓
医生电话回拨，指导紧急处置
       ↓
宠物到院，转为线下处理
       ↓
医生录入处理记录，关闭 SOS
```

---

## 二、用户端 SOS 流程详细设计

### 2.1 第一步：触发 SOS

**页面**：首页

**设计要点**：
- 首页核心是一个大尺寸 SOS 按钮
- 颜色：紧急红色（`#E53935`）
- 图标：🆘
- 文字：「立即求助」
- 点击后**直接进入流程**，不弹出确认对话框（节省时间）
- 首次使用时需要宠物档案和位置授权，建议在非紧急状态下完成

**防误触机制**：
- 按钮需长按 1 秒才触发（可配置，避免误触）
- 或采用双击触发

---

### 2.2 第二步：快速填写症状（核心问题）

**页面**：SOS 问卷页

**设计原则**：
- 总问题数 ≤ 5 个，力求在紧急状态下尽快完成
- 每题均有快速选项（单选/多选），同时支持文字补充
- 界面字体大，操作区域大，减少手误

**必填问题清单**：

| # | 问题 | 答题形式 | 选项示例 |
|---|------|----------|----------|
| 1 | 主要症状是什么？ | 多选 + 文字 | 呕吐 / 抽搐 / 呼吸困难 / 出血 / 昏迷 / 其他 |
| 2 | 症状持续多久了？ | 单选 | 刚刚发生 / 10分钟内 / 30分钟内 / 超过1小时 |
| 3 | 宠物现在的意识状态？ | 单选 | 清醒有反应 / 昏迷无反应 / 不确定 |
| 4 | 是否有明显出血或外伤？ | 单选 | 是 / 否 / 不确定 |
| 5 | 请选择就近医院 | 地图选择 | 系统推荐附近医院列表 |

**红旗关键词触发规则**（后台逻辑，用户不可见）：

以下情况系统自动标记红旗（🚩）：
- 意识状态选择"昏迷无反应"
- 症状包含"抽搐""呼吸困难""出血"
- 症状持续时间超过 30 分钟

---

### 2.3 第三步：确认自动补全信息

**页面**：信息确认页

**自动补全内容**（只读展示）：

| 字段 | 来源 |
|------|------|
| 联系电话 | 用户注册信息 |
| 当前位置 | GPS 定位（显示地址文字） |
| 宠物姓名 | 宠物档案（已绑定） |
| 宠物种类/品种 | 宠物档案 |
| 提交时间 | 系统时间戳 |

**用户操作**：
- 如有宠物档案选择（多只宠物），在此页选择当前救援的宠物
- 如位置不准，支持手动调整

---

### 2.4 第四步：确认并提交

**页面**：提交确认页

- 展示所有填写内容的摘要
- 一个大「确认提交」按钮
- 提交后立即显示等待页，告知医院将在 X 分钟内回电

---

### 2.5 第五步：等待医生回拨

**页面**：等待页

- 显示"已成功发送至 [医院名称]"
- 显示"请保持手机畅通，医生将在 5 分钟内联系您"
- 显示紧急自助提示（不含医疗诊断）：
  - "保持冷静，不要随意移动宠物"
  - "如宠物有呕吐，侧卧防止窒息"
  - "记录症状变化，便于告知医生"
- 显示 SOS 编号（用于医院查询）

> ⚠️ 等待页提示内容须经执业医生审核，不得包含任何诊断或药物建议。

---

## 三、医院后台 SOS 处理流程

### 3.1 接收 SOS

- 新 SOS 到达时，后台实时推送通知（声音 + 视觉提醒）
- 红旗 SOS 排列在列表顶部，显示红旗图标（🚩）
- 列表信息：SOS 编号 / 宠物名 / 主要症状 / 提交时间 / 距离 / 状态

### 3.2 分配与处理

| 步骤 | 操作 | 负责人 |
|------|------|--------|
| 接单 | 接诊员确认接单，状态变为"处理中" | 接诊员 |
| 分配 | 分配给对应科室医生 | 接诊员 |
| 回拨 | 医生查看详情后拨打联系电话 | 医生 |
| 指导 | 电话中指导紧急处置（不做诊断，仅做应急处置） | 医生 |
| 记录 | 录入回拨时间和指导要点 | 医生 |

### 3.3 到院后处理

- 宠物到院后，SOS 状态更新为"已到院"
- 医生继续在系统内补充处理记录
- 治疗完成后，状态更新为"已关闭"
- 系统自动生成病例摘要（供医生核准存档）

## 四、数据采集与沉淀

### 4.1 结构化数据字段

每条 SOS 记录包含以下结构化字段，便于后续分析：

```
- SOS 触发时间
- 提交完成时间（计算填写耗时）
- 医院接单时间
- 医生回拨时间
- 宠物到院时间
- 最终处理结果
- 症状关键词列表
- 红旗标记及原因
- 医生指导摘要
```

### 4.2 脱敏规则

用于分析和未来 AI 训练的数据须经以下脱敏处理：

| 字段 | 脱敏方式 |
|------|----------|
| 手机号 | 保留前3位，其余用 `****` 替代 |
| 用户姓名 | 以 `用户_XXXX` 替代 |
| 精确定位 | 模糊至区/街道级别 |
| 宠物名称 | 保留（非个人隐私） |

### 4.3 数据用途说明

- 当前阶段：运营分析，提升响应速度
- 授权后：脱敏数据用于 AI 模型训练（需用户明确授权）
- 严禁：未授权共享给第三方，未脱敏用于 AI 训练

## 五、紧急情况兜底机制

即使系统出现故障，也必须保证以下兜底机制：

1. **离线缓存**：SOS 数据本地缓存，网络恢复后自动上传
2. **电话兜底**：等待页显示医院急救电话，供用户直接拨打
3. **短信通知**：SOS 提交后同时向医院发送短信备份
4. **超时报警**：SOS 提交后 5 分钟未被接单，系统自动向所有在线接诊员发出报警

---

## 六、未来 AI 辅助能力预留

以下能力在当前 MVP 阶段**不实现**，但数据结构和接口须为其预留扩展点：

| 未来能力 | 说明 |
|----------|------|
| 症状预筛选 | 基于历史数据推荐最可能的症状组合 |
| 智能分诊建议 | 辅助接诊员判断优先级（仍需人工确认） |
| 病例摘要生成 | 自动整合 SOS 记录生成结构化摘要 |
| 响应时间预测 | 预测最快响应医院 |

> ⚠️ 以上所有 AI 能力在上线前必须通过医疗安全审查，且只能辅助医生，不能替代医生判断。

# 计划模板

复杂任务开始前，先按本模板写计划。计划必须足够明确，让另一个工程师或 AI agent 可以直接执行，不需要再做关键决策。

## 任务标题

用一句话说明任务，例如：

```text
初始化小程序工程地基
```

## 目标

说明这次任务要达成什么结果。

示例：

- 建立可运行的 Taro + React + TypeScript 小程序工程。
- 不写业务页面，只验证启动、测试和构建链路。
- 保持医疗安全边界和现有文档规则不变。

## 范围

本次包含：

- 需要新增或修改的模块。
- 需要新增的配置、脚本或文档。
- 需要验证的命令。

本次不包含：

- 明确排除的功能。
- 不允许触碰的目录。
- 不允许引入的依赖或技术栈。

## 当前状态

记录开始前的事实：

- 当前分支。
- `git status --short` 摘要。
- 当前是否有未提交改动。
- 当前是否已有启动、测试、构建命令。

## 实施步骤

按顺序列出可执行步骤：

1. 创建或确认专门分支。
2. 修改文档或代码。
3. 更新相关说明。
4. 运行验证命令。
5. 更新 `更新记录.md`。

## 验收标准

用可验证语言描述完成标准：

- 文件是否存在。
- 命令是否通过。
- 哪些关键词必须保留。
- 哪些旧入口必须不存在。

## 测试计划

列出要运行的命令和预期结果。

当前工程骨架阶段如果没有脚本，应明确写：

```text
当前没有可运行测试脚本，本次只做文件结构和内容校验。
```

## 风险与处理

必须列出：

- 医疗安全风险。
- 隐私和密钥风险。
- Git 分支风险。
- 真实医院数据风险。
- 依赖安装风险。

## 回滚方案

说明如何撤销本次工作：

```bash
git switch main
git branch -D feature/<short-task-name>
```

如果已提交并共享，使用：

```bash
git revert <commit-sha>
```

不得默认使用 `git reset --hard`。

## 交接记录

完成后把以下内容写入 `更新记录.md`：

- 改了什么。
- 当前状态。
- 下一步。
- 风险。
- 涉及文件。
